4.1 Ansible Playbook

A LAMP stack is web application stack that is used to create websites using some of the most used technologies. The following are the four technologies that are collectively known as the 'LAMP':

  • Linux: The operating system for the server being used.
  • Apache: Apache is the web server which is used for creating the application.
  • MySQL/MariaDB: The database to be used which can aid in feeding the data to the web application
  • PHP: The language used to create the logic/pages of the websites.

In this section, we will be creating a LAMP stack to host a website on the Node1. This playbook is going to perform the following plays:

  • Install packages on the node such as Apache server (HTTPD), MariaDB, PHP
  • Enabling HTTP services on the node such that the example website is accessible on the Internet. (Remember we set the HTTP protocol in our Security Group, that enabled the inbound rule for accepting the HTTP requests)
  • Start Apache's HTTPD web server
  • Start MariaDB database server
  • Download a sample PHP page from a remote URL
  • Access the website through this PHP page

4.1.1 Ansible configuration file

First we need to look at and update the ansible configuration file.

  • Configuration files in your cloned Git repo

    $ cd ~/ansible-demo/lamp
$ cat ansible.cfgcd ~/ansible-demo/lamp
cat ansible.cfg

    Output:

    [defaults]
host_key_checking = False
inventory= /home/ec2-user/ansible-demo/inventory/hosts
remote_user = ansibledevops
private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"

    Notice that this configuration file is almost the same to the previous one that we used in Chapter 3, with only one difference: the remote_user is now changed to "ansibledevops". This is because we have configured a new user on the remote nodes and disabled ec2-user from logging into the nodes. From now on, the remote user you will be using on the remote nodes will be just the "ansibledevops".

  • Update the Ansible configuration

    Update the "privatekeyfile" variable to use your own private key by changing "YourPrivateKey" to your own value. In the below command, change the 2nd "YourPrivateKey" to your own private key name.

    $ sed -i 's&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&g' ansible.cfg$ sed -i 's&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&g' ansible.cfg

    For example, in my case (my private key is "id_rsa"): 

    $ sed -i 's&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&private_key_file = /home/ec2-user/.ssh/id_rsa&g' ansible.cfgsed -i 's&private_key_file = /home/ec2-user/.ssh/"Your_Private_Key"&private_key_file = /home/ec2-user/.ssh/id_rsa&g' ansible.cfg

    Or, if the command above is confusing you, just do the following: 

    $ vim ansible.cfgvim ansible.cfg

    And manually update the value "YourPrivateKey" in the last line to the name of your private key.

4.1.2 Check Playbook file

We have prepared the Playbook yaml file for this example application.

  • Check the Playbook file: 

    $ cd ~/ansible-demo/lamp
$ cat lampStack.yamlcd ~/ansible-demo/lamp
cat lampStack.yaml

    Output:

    YAML
    ---
- name: LAMP for website
  user: ansibledevops 
  hosts: node1 
  become: yes
  tasks:
    - name: install related packages 
      yum:
        name:
          - httpd
          - mariadb-server 
          - php
          - php-mysql
        state: latest
    - name: enable and start HTTP service
      service:
        name: httpd 
        enabled: true 
        state: started
    - name: enable and start mariadb 
      service:
        name: mariadb 
        enabled: true 
        state: started
    - name: get index file 
      get_url:
        url: "https://www.middlewareinventory.com/index.php" 
        dest: /var/www/html/index.php
        mode: 0644
      check_mode: no
    - name: view webpage 
      uri:
        url: http://Node1_Public_IP/index.php 
        status_code: 200
      check_mode: no

    Use the following content in the yaml file. Observe the YAML file will define the following things:

    • user - which user to use, for sudo purposes or as different settings.
    • name - sets the name of the plays
    • hosts - mostly used as a group of hosts defined in the inventory file. For our example, we will just pick the node1 as host.
    • become - gives flexibility for the engine to become certain user such as e.g., a root user
    • tasks - are just the different tasks that are streamlined. Note that tasks as 'yum', 'service' etc. are linux commands that the playbook is going to execute.
    • Check_mode: no - specifies that during dry-run these tasks are not going to be executed since they have dependencies on previous tasks which do not get executed during dry-run as well.

  • Update the Playbook file You will need to update the Playbook to access the web page after installing the LAMP stack on your node1. Run the following command to update the "url" by replacing "YourNode1Public_IP" with your own node1 public IP: 

    $ sed -i "s&url: http://Node1_Public_IP/index.php&url: http://Your_Node1_Public_IP/index.php&g" lampStack.yaml

    For example, in my case: 

    $ sed -i "s&url: http://Node1_Public_IP/index.php&url: http://44.200.9.15/index.php&g" lampStack.yaml$ sed -i "s&url: http://Node1_Public_IP/index.php&url: http://44.200.9.15/index.php&g" lampStack.yaml
