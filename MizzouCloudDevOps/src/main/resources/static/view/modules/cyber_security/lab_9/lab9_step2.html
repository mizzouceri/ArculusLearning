1.1 Purpose of the Lab

The purpose of the lab is to help deploy machine learning models on cloud platforms in a secure way, ensuring that both the model and data are protected from external access, while also maintaining the integrity of computations when the data is in use. This is achieved using Confidential Computing — a technology that enables data to be processed in a protected environment, also known as a Trusted Execution Environment (TEE). These environments isolate sensitive data and code from the rest of the system, even from cloud providers or malicious insiders, ensuring that computations remain private and unaltered. For instance, in bioinformatics, machine learning models are often used to analyze sensitive genomic data. Deploying such models securely on the cloud ensures that patient information remains confidential while enabling large-scale computations without compromising data privacy.

In this lab, you will get hands-on experience with the Certifier Framework, an open-source project that simplifies the deployment of AI applications within a confidential computing environment. The framework allows users to define, enforce, and verify security policies, perform AI application attestation for trusted communication between components in a cloud-native setup. Attestation means a unique hashcode generated for an application based on its program. This attestation prevents forging of the application and hence preserves privacy and integerity of the application

1.2 Prerequisites

To successfully follow this lab, it is important to understand the key components of the Certifier Framework. These include the Certifier API and the Certifier Service, which form the foundation of secure communication and trust management within the framework. The Certifier API is a lightweight, client-side library that handles attestation of AI application and secure storage. It abstracts the complexity of secure communication and helps client application to establish trust with remote certifier services. The Certifier Service acts as the central policy evaluation engine (users can define rules embedded as policies). It enables scalable, trust-based management of applications by enforcing policies and validating application attestation reports. This service ensures that only trusted components are allowed to interact with each other in the system. A basic understanding of how these components interact will help you navigate the lab exercises more effectively. If you're new to the Certifier Framework, we recommend reviewing the introductory materials or previous labs before proceeding.

1.4 Overview

The architecture below (figure 1.1) illustrates how the Certifier Framework facilitates secure interactions between distributed applications using trusted execution environments (TEEs) and centralized policy management. At the core is the Certifier Service, which defines the domain policies, trust rules, and service configurations. It communicates with client applications to verify their integrity and compliance with the security policies. Each application integrates the Certifier Library, which supports attestation, secure storage, and trusted communication.


Figure 1.1: Overview of the Certifier Framework Architecture.


The main features include:

  • A policy-driven trust model using cryptographic identities and signed security rules.
  • Lightweight client-server architecture with strong attestation and verification support.
  • No reliance on distributed consensus; individual nodes are independently trusted.
  • Secure interactions are based on verified code and platform states.
  • Policy provisioning and updates are securely handled during runtime.
  • Integration support with cloud, edge, and confidential computing environments.
  • Compatible with diverse deployment models and DevSecOps workflows.
