So far, our Maven project can build the application, run unit tests, and build the artifact. The next step in our continuous integration process would be to actually deploy the artifact to staging environment to test it before finally deploying the application to production.

In this chapter, we will archive the war artifact generated in our Maven project, and deploy it to a staging environment using Apache Tomcat web server. On the AWS EC2 AMI image, we have installed and configured Tomcat server, added a Tomcat user and configured Tomcat to run on Port 9090 on the VM instance. In this case, we are creating a Jenkins pipeline that seamlessly integrates and automates CI and CD processes starting from GitHub repository to staging environment.

3.1 Configure CI part of the pipeline.

From the maven-job project page, click “Configure”. Scroll down to “Post-build Actions”, and from the drop-down menu select “Archive the artifacts”. In the “Files to archive”, input “**/ *.war”, and then “Save”.

From the maven-job project page, click “Build Now”. Once the new build is done, click the build , and check the “Status”. As you can see, the build has generated war Artifacts as shown in the below screenshot.

Change the name of the maven-job project to avoid confusion with the CD project you will setup in the next step. This can be done by go to the maven-job project page and clicking “Rename” and then change it to “packaging” or something else so that you can easily know that it is the first step of your process, i.e. integration step.

3.1.1 Install required plugins to Jenkins.

From Jenkins Dashboard page, click “Manage Jenkins”, then click “Plugins”. In the “Available Plugins” tab, search for “Copy Artifact” Plugin. Once you find it, check the box and select “Install” to install this plugin. Repeat the same process for another plugin, “Deploy to Container” Plugin.

3.2 Configure CD part of the pipeline

From the Jenkins Dashboard, click “New Item” to create another project. Name it as “deploy-to-staging” or whatever you like, and select “Freestyle project”.

Leave all setting as default and scroll down to “Build”. Click “Add build step” and select “Copy artifacts from another project”. In the “Project name”, choose “packaging” or the name you specified for your first maven project earlier. And in the “Artifacts to copy”, add “**/ *.war”.

Scroll down to “Post-build Actions”, click “Add post-build action”, and then select “Deploy war/ear to a container”. In the “WAR/EAR files”, add “**/ *.war”.

The post-build deployment process needs a server to deploy the artifact, and also needs credentials to remotely operate on the Tomcat server. Thus you will need to input the credentials for a remote Tomcat user. During our pre-installation, we have created a user/password for you. Under the “Containers”, click “Add Container”, and select the latest version of the Tomcat, in my case, the “Tomcat 9.x Remote”. Although we are using Tomcat 10 in our Docker container, Tomcat 9.x should also work for this process.

Then In the “Credentials”, click “Add” and select “Jenkins” to add a credential. From the pop up window, add the user “tomcat” and password “jenkins-demo”. Click “Add” and return to the main page.

Under the “Credentials”, click the drop down and select “tomcat/****” that you just added. In the Tomcat URL, add your EC2 instance IP with the port 9090 (http://EC2_IP:9090). Finally, click Save.

3.2.1 Add upstream trigger to build staging job

Your staging job has been configured, but it does not know when to build to copy and deploy the .war file. To make it aware of its upstream job and activate automatic building process, you will need to configure the “packaging” job to activate the “staging” job. From your “packaging” job main page, click “Configure”. Scroll down to the “Post-build Actions”, and click “Add post-build action”. Choose “Build other projects”, add the name of the staging job, i.e. “staging” into the projects to build, and leave the trigger as “Trigger only if build is stable”. Click “Save” to finish configuration.

3.2.2 Start Tomcat server on the VM instance

Although we pre-installed Tomcat server on the EC2 machine image, it will not start unless you manually do it. Run the following command on your terminal console to bring up Tomcat server:

$ sudo /usr/local/tomcat/bin/startup.sh

3.3 Test your pipeline manually

Based on your pipeline settings, your “deploy-to-staging” project will be triggered by your “packaging” project automatically. Let’s test it!

3.3.1 Run your pipeline.

From Jenkins Dashboard page, build your “packaging” project by clicking the green play button.

Immediately you will see that the “packaging” project has been scheduled for a build under “Build Executor Status”. Then shortly it will finish, and the “deploy-to-staging” is added into the “Build Queue” for its build. Again, it will complete in a couple of seconds. And you will see that there is a green check mark on the left side of the “deploy-to-staging” project.

Click the name of “deploy-to-staging” project, and explore the build information. You will see that under the “Build History” there is the build that was triggered by the “packaging” project.

Click the build # to view the details of the build. You will see that the “deploy-to-staging” project has a dependency project, “packaging”, and the current build was started by the “packaging” project from an “Upstream Build” from that project.

Click the “Console Output” tab from the left. You will see the details of the current build of the “deploy-to-staging” project.

3.3.2 Access the web server

Once the build successfully completed, go to your web browser, and access the server using the URL “http://VM_IP:9090/webapp/”. You will be able to see the greeting message from the server as below, indicating that your CD project has been successfully deployed to the Tomcat server.

3.4 Create a Jenkins pipeline

So far we have created a Jenkins build pipeline which consists of two jobs. An upstream job which is the package job and a downstream job which is ”deploying to stage” job. Jenkins lets you build multi-step build pipelines and trigger the rebuild of a project if one of its dependencies are updated. Currently we only have two jobs in this build pipeline so it is very clear that how those two jobs are connected. However, In real life scenarios, we might have dozens of builds jobs or even hundreds especially for some complicated Jenkins build pipelines. It would become more and more difficult for us to understand the pipeline as the number of build jobs increases. We need a good way to visualize the build pipeline to understand how those jobs are connected. This is where the Jenkins build pipeline plugin comes into handy. The build pipeline plugin offers a build pipeline view of upstream and downstream connected jobs that typically form a build pipeline. By configuring a new Jenkins view and choosing which job is the first job in the Jenkins pipeline you can get a visualization of the whole build pipeline.

3.4.1 Install Build Pipeline plugin

Go to your Jenkins Dashboard page, and click “Manage Jenkins” and then click “Plugins”. From the “available plugins” search for “Build Pipeline” plugin. You probably can notice that there is a warning message about potential XSS vulnerability. Just ignore it and install the plugin without restart:

3.4.2 Create a new pipeline view

Now let’s create a new view of your pipeline. View is a good way to help you to visualize your pipeline structure, and thus to better organize your jobs. You could create a view with a subset of your Jenkins project. Click “New View” from the left side of your Jenkins Dashboard page. In the view page, give it a name you like.

The only place you need to configure in the view is the “Pipeline Flow” section where you need to specify the “Upstream” with the “Initial Job”. You need to select the “packaging” project from where you created before (or whichever name you specified for your CI process).

Your pipeline view will display all the processes in your pipeline. Now you only have two of them in the pipeline, both having a green background since we have run them before:

3.4.3 Run the pipeline

From the pipeline view page, click the green “Run” button to test your pipeline configuration. You will first notice that the “packaging” project is initiated and it turns into yellow background with a process bar indicating it’s running. And the “deploy-to-staging” job has a blue background indicating it’s currently not active. After a little while, the “packaging” job is done, and the “deploy-to-staging” is in the queue and then also gets run. At the end, both projects are run and turn back to green again, and the pipeline has finished running.

If your packaging project still has “GITScm polling” configured, you can update your GitHub repo and see if it can trigger your pipeline automatically!
