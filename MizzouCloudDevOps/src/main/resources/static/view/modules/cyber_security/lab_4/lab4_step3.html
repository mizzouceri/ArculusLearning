2.1.2 Create an EC2 VM instance

Access AWS EC2 Service

  • Login to your Amazon AWS cloud account,
  • Make sure to select the US East (N. Virginia) region in the top-right part of your screen. Then click “Services” on the top left corner and select “EC2” service (Elastic Compute Cloud) under the “Compute”.

Manage Key Pairs

  • In left menu, select “Key Pairs” under “Network Security” option.

  • Click on “Create Key Pair” and create a key pair with a name you like, type ”RSA”, and pem/ppk based on your SSH client type (pem for Mac and ppk for Win10 with PuTTY). Store your downloaded .pem file in a safe location, you will need this file to login to your EC2 instance.

Manage Security Group

  • Select “Security Groups” from the left menu, and click ”create security group”. Give it a name that you like, add any description, and appropriate inbound/outbound rules to the security group. Make sure to allow your computer’s IP to connect to the EC2 instance. For the lab purpose, you can set the inbound/outbound rules to allow all traffic/protocol/ports and set the Source IP range to 0.0.0.0/0. Click on Create security group at the bottom

Launch an Instance

Installing Jenkins on AWS EC2 instance requires a number of pre-requisite libraries, e.g., Docker, Docker-compose, Git, Java, Maven, Tomcat, etc. For this lab, we have prepared an AWS machine image (AMI) that has these libraries installed. You will just use the AMI to launch new instances, and continue with the rest of the lab procedure.

  • From the left-side menu, click the “AMIs” tab under “Images”
  • On the right-hand side, select “Public images” from the drop down menu, and search for our prepared image using the image ID “ami-04ddbbc708819dc29”. It should shows you an image with the name “mizzou_jenkins_image”;
  • Once the image shows up, click the image ID and then “Launch instance from image”.

  • In ‘Name’ section, give a name to the instance;

  • In‘Choose Instance Type’ section, select the t2.medium instance; ‘Choose Key pair’, select the key pair you previously created;

  • In ‘Select existing security group’ section, select the security group you previously created;

  • In ‘Configure storage’ section, give the storage size as 30GB. Click on ‘Launch instance button’;

  • Click on ‘View all instance’, you will be redirected to the instances page;

- In a short time your new instance will be deployed and ready for you to use. Note down the “Public IPv address” of the VM instance in the “Details”. You will need the IP address to login to the VM instance.

2.2 Jenkins installation

In this lab, we will install Jenkins using the Linux version. It can also be installed using the Docker version, however we found that it will be quite complicated to deploy our example use case application, which is also Docker-based, later in the lab. Jenkins can be installed on the EC2 instance by following this link:Jenkins Installation on Ubuntu.

2.2.1 Login to the AWS EC2 instance

Selecting your instance, click on the "Connect" button available at the top nav bar. Replace the username to "ubuntu".

Alternatively, you can ssh into the instance from your local machine. First, change the permissions on the downloaded key:

If you are using Mac/Linux OSes, run:

$ chmod 600 YOUR_KEY

For Windows user, run the following on powershell/cmd:

$icacls jenkins-key.pem /inheritance:r
$icacls jenkins-key.pem /grant:r "${env:UserName}:R"

AWS EC2 instance using Ubuntu as machine image requires you to login to the VM instance using the user ”ubuntu”. Login to the VM with the following command from the directory that contains your SSH key:

$ ssh -i YOUR_KEY ubuntu@VM_public_IP

2.2.2 Install Jenkins.

On Debian and Debian-based distributions like Ubuntu you can install Jenkins through apt. Follow the below commands to install Jenkins:

First of all, let us install Java 17 in order to run Jenkins:

$ sudo apt install openjdk-17-jdk -y

Next, collect required libraries for installing Jenkins. Run these codes one by one:

$ wget -q -O /usr/share/keyrings/jenkins-keyring.asc https://pkg.jenkins.io/debian/jenkins.io-2023.key
$ echo "deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] https://pkg.jenkins.io/debian-stable binary/" | sudo tee /etc/apt/sources.list.d/jenkins.list > /dev/null
$ sudo apt update
$ sudo apt install jenkins

Register the Jenkins service with the command: 

$ sudo systemctl daemon-reload
You can start the Jenkins service with the command:

$ sudo systemctl start jenkins

You can check the status of the Jenkins service using the command:

$ sudo systemctl status jenkins

If everything has been set up correctly, you should see an output like this:

● jenkins.service - Jenkins Continuous Integration Server
  Loaded: loaded (/lib/systemd/system/jenkins.service; enabled; vendor preset: enabled)
  Active: active (running) since Wed 2022-04-06 16:39:54 UTC; 34s ago

2.2.3 Post-installation setup wizard

Jenkins is installed, and it’s up and running. You will now need to setup the Jenkins through its web portal by following the steps below. This setup wizard takes you through a few quick steps to unlock Jenkins, customize it with plugins and create the first administrator user through which you can continue accessing Jenkins.

Unlocking Jenkins

When you first access a new Jenkins instance, you are asked to unlock it using an automatically-generated password.

  • Browse to http://HostVMIP:8080 (Replace the HostVMIP with the public IP (IPv4) of the AWS EC2 instance). If you configured the Jenkins with a different port, change 8080 to the actual port you are using.

  • From the Jenkins console log output, copy the automatically-generated alphanumeric password (between the 2 sets of asterisks). Run the command:

    $ sudo cat /var/lib/jenkins/secrets/initialAdminPassword

  • On the Unlock Jenkins page, paste this password into the Administrator password field and click Continue.

    Note: The Jenkins console log indicates the location (in the Jenkins home directory) where this password can also be obtained. This password must be entered in the setup wizard on new Jenkins installations before you can access Jenkins’s main UI. This password also serves as the default admininstrator account’s password (with username ”admin”) if you happen to skip the subsequent user-creation step in the setup wizard.

2.2.4 Customizing Jenkins with plugins.

After unlocking Jenkins, the Customize Jenkins page appears. Here you can install any number of useful plugins as part of your initial setup. Click one of the two options shown:

  • Install suggested plugins - to install the recommended set of plugins, which are based on most common use cases.
  • Select plugins to install - to choose which set of plugins to initially install. When you first access the plugin selection page, the suggested plugins are selected by default.
Note: If you are not sure what plugins you need, choose Install suggested plugins. You can install (or remove) additional Jenkins plugins at a later point in time via the Manage Jenkins >Manage Plugins page in Jenkins.

2.2.5 Creating the first administrator user

After customizing Jenkins with plugins, Jenkins asks you to create your first administrator user.

When the Create First Admin User page appears, specify the details for your administrator user in the respective fields and click Save and Finish.

When the Jenkins is ready, click Start using Jenkins

Note: If the page does not automatically refresh after a minute, use your web browser to refresh the page manually.

Log in to Jenkins with the credentials of the user you just created and you are ready to start using Jenkins!

2.3 Jenkins UI Overview

Once you login to the Jenkins portal, you are at the main landing page or dashboard.

On the left hand side of the dashboard, you will the following tabs:

  • New Item: the item refers to a job or a project. It helps us to create a new job.
  • People: this menu item manages users within Jenkins.
  • Build History: shows us a history of builds in this Jenkins instance. It includes the builds that were executed not only on the master node but also on the slave nodes.
  • Manage Jenkins: This configure System item can help us to configure global settings and paths, such as the path to Maven and Java, security settings, system information and system logs, etc.
  • My views: This displays a private view which provides a possibility for every user to have its own filter of Jenkins Jobs.

  • Open Blue Ocean: to switch to Blue Ocean, a new user interface for Jenkins. Blue Ocean makes Jenkins, and continuous delivery, approachable to all team members.

  • Lockable Resources: allows defining lockable resources (such as printers, phones, computers, etc.) that can be used by builds. If a build requires a resource which is already locked, it will wait for the resource to be free.

  • New View: allows you to create a customized view of the items in a simple list format or displays all the jobs that the you have access to.

  • Build Queue: shows all the builds that have been triggered.

  • Build Executor Status: displays the current build executors on this Jenkins instance.

On the right hand side, there are:

  • A welcome message: “Welcome to Jenkins!”
  • Start building your software project: for creating a Jenkins job.
  • Set up a distributed build: to create a distributed job on a cluster.

Adding automate build triggers to the project

Go to the Maven project home page, and click “Configure”. Scroll down to the “Build Triggers” section. From this section, you will see a few options as triggers as follows:

  • Trigger builds remotely: Jenkins provides machine-consumable remote access API to its functionalities. This trigger allows you to trigger a build by using the Jenkins API URL to manually trigger the build.
  • Build after other projects are built: This option lets you set up a build that will be run whenever another build has finished. This is an easy way to set up a build pipeline.
  • Build periodically: This allows you to trigger the build job at regular intervals.
  • Build GitHub hook trigger for GITScm polling: This feature enables builds after post-receive hooks in your GitHub repositories. Basically instead of Jenkins polls SCM every minute, what happens here is that whenever a commit is made to Github, a web hook will be triggered which will notify Jenkins to kick off a build right away.
  • Poll SCM: This option allows you to configure Jenkins to pull the source code periodically, and triggers a build whenever it sees any changes in the source code. With this option, all you need to do is to specify a schedule, e.g. every hour or every day, for Jenkins to check for source code changes and trigger builds. This options has huge overhead as every polling requires Jenkins to scan the entire workspace and verify it with the server for any source code changes.

In this lab, we will try one of the options, the “Build GitHub hook trigger for GITScm polling” option. From the “Build Triggers”, check the “Build GitHub hook trigger for GITScm polling” option”, and click “Save”.

From your github account, go to your cloned “jenkins-demo” repository. From the repo root page, click “Settings”, and then “Webhooks”. Then click “Add webhook”. Use the settings from the screenshot below by replacing the “your_IP” with your own IP address running the Jenkins server. And click “Add webhook”.
